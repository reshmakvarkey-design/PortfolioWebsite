Role Senior Product Designer
Type AI Agent, Enterprise

Overview

Design an AI-powered agentic system that democratizes privacy expertise, enabling data engineers to independently conduct risk-based anonymization while maintaining regulatory compliance and maximizing data utility.

This case study showcases the design of an AI-guided anonymization experience for pharma data teams, guide users through the complex decision-making process of risk assessment, strategy selection, and validation.

This case study showcases the design of an AI-guided anonymization experience for pharma data teams, turning a complex, risky workflow into an AI-orchestrated, human-in-the-loop experience. The design emphasizes how the AI agent guides, explains, and collaborates with experts through high-stakes decisions.

Problem Statement

Pharmaceutical organizations struggle to share sensitive patient datasets safely and efficiently while maintaining regulatory compliance and data utility. Existing tools provide technical anonymization functions but don't guide users through the complex decision-making process.

The Pain Points

  • Expertise Barrier: Data engineers lack specialized privacy domain knowledge; every request requires privacy officer review.
  • Process Inefficiency: 2-4 hours per dataset, multiple tool switches, 40% abandonment rate.
  • Privacy-Utility Trade-off: Find the optimal balance between analytic utility and privacy compliance.
  • Inconsistent Methodology: Lack of standardized risk assessment across teams.
  • High-Stakes Consequences: severe penalties for violations and risks to patient privacy.

User Roles and Needs

Data Engineer

  • Prepares and uploads clinical or operational datasets for anonymisation, ensuring schema and metadata are correct.

Transparency Team

  • Owns disclosure and anonymization of data for publications, regulatory transparency, and external sharing.
  • Uses the platform to assess re‑identification risk, approve anonymisation strategies, and ensure outputs meet transparency and compliance requirements.
  • Collaborates with the agent to confirm quasi‑identifiers, review anonymisation rules, and verify that analytics usability is preserved.

Data Governance Lead / Research Data Manager

  • Defines policies for acceptable risk thresholds and sharing contexts (internal, external, public) and configures them in the platform.
  • Monitors all anonymisation requests, reviews audit trails, and ensures that data sharing aligns with organisational governance and regulatory standards.

UX Approach

  • Discovery: Mapped the current journey, interviewed data engineers and transparency teams.
  • Definition: Defined key workflows and the data transaction between user and agents.
  • Design: Designed a conversational, stepwise flow orchestrated by an AI agent.
  • Validation: Ran walkthroughs with target users using realistic datasets.

Persona

Proposed Solution

A platform adds intelligent pre-processing: profile data, calculate risks, incorporate context, then invoke the existing service with tailored rules.

  • Dataset upload and automated profiling for quasi-identifiers (e.g., age, location, rare codes).
  • User sets sharing context and risk threshold via sliders/selector.
  • AI agent computes re-identification risk with visual gauge (green/yellow/red).
  • Generate optimised anonymisation rules explaining privacy-utility trade-offs.
  • Invoke existing service with rules, followed by before/after verification and audit log.

Step 1: Request Initiation

Actor: User

Action: User initiates anonymization request by providing Dataset ID

System: Agent acknowledges request and explains the workflow

Agent Message: "Hello! I'm here to assist you with risk-based anonymization. I'll guide you in calculating data risk and suggest strategies to mitigate those risks."

Step 2: Data Selection

Actor: Agent

Action: Agent detects available data folders/files for the dataset

System: Displays available options with metadata

Actor: User

Action: User selects target data to anonymize

Step 3: Quasi-Identifier Detection

Actor: Agent

Action: Agent analyzes selected data and automatically detects quasi-identifiers

System: Presents detected identifiers with metadata:

  • Age (demographics table)
  • Geographic Location (demographics table)
  • Medical Condition Code (health records table)

Actor: User

Action: Reviews and accepts detected quasi-identifiers

Button: "Accept"

Step 4: Data Sharing Context Configuration

Actor: Agent

Action: Agent requests data sharing context parameters

System: Displays configuration options:

  • Sharing scope: Internal Use / External-Collaborator / Public
  • Access type: Secure Portal / Non-Secure Portal

Actor: User

Action: Selects appropriate sharing context

Agent Response: "Can you please provide the data sharing scope and sharing portal details"

Step 5: Risk Threshold Configuration

Actor: Agent

Action: Prompts user to set acceptable re-identification risk threshold

System: Displays slider interface (range: 0 to 1)

Actor: User

Action: Sets risk threshold based on data sensitivity and use case

Default Value: 0.2 (configurable based on organizational policy)

Button: "Proceed"

Step 6: Automated Risk Calculation

Actor: Agent

Action: Executes risk calculation based on detected quasi-identifiers and context

System: Computes re-identification risk using statistical methods

Display: Risk gauge visualization showing:

  • Calculated Risk: 0.21
  • Risk Threshold: 0.20
  • Status: EXCEEDS THRESHOLD (risk mitigation required)

Options:

  • "View Risk Report" - Detailed risk analysis
  • "View Sample Data" - Preview of data structure
  • "Accept" or "Show Risk Mitigation"

Step 7: Risk Report Review

Actor: User

Action: Reviews comprehensive risk assessment report

System: Displays:

  • Individual quasi-identifier risk contributions
  • Combined re-identification probability
  • Data distribution statistics
  • Risk factors and vulnerabilities

Actor: User

Decision: Accept current risk level OR request mitigation strategies

Button: "Accept" (if risk ≤ threshold) or "Show Risk Mitigation" (if risk > threshold)

Step 8: AI-Generated Anonymization Recommendations

Actor: Agent (triggered when risk exceeds threshold)

Action: Analyzes risk report and autonomously generates anonymization strategy

Agent Message: "Any additional context I need to consider before calculating risk of re-identification and proposing anonymization options"

System: Presents recommended anonymization functions:

  • Age: Keep (low risk contributor)
  • Geographic Location: Change to regional level (generalization)
  • Medical Condition Code: Remove (high risk, low analytical value)

Actor: User

Action: Reviews proposals; can modify or accept recommendations

Option: Add custom anonymization rules via "+" button

Step 9: Data Utility Loss Assessment

Actor: Agent

Action: Calculates expected data utility loss from proposed anonymization

Agent Message: "Please find the method to calculate data utility loss. Do you want to continue with re-anonymization?"

System: Displays utility loss metric with methodology link

Result: "Data Utility Loss: 0.23" (23% information reduction)

Actor: User

Action: Evaluates privacy-utility trade-off

Button: "Calculate Risk" (to preview) or "Proceed" (to accept)

Step 10: Re-Anonymization Execution Authorization

Actor: Agent

Action: Requests final authorization to execute re-anonymization

Agent Message: "Would you like me to provide more details on the selected option or proceed with implementing the anonymization rules?"

System: Displays execution summary with:

  • Selected anonymization rules
  • Expected final risk level
  • Data utility impact
  • Output dataset location

Actor: User

Action: Authorizes execution

Button: "Re-anonymize"

Step 11: Autonomous Agent Execution

Actor: Agent

Action: Applies anonymization rules to dataset autonomously

System:

  • Executes transformation pipeline
  • Applies generalization, suppression, and removal operations
  • Creates new anonymized dataset
  • Preserves original data unchanged

Progress: User sees status updates during execution

Step 12: Post-Anonymization Verification

Actor: Agent

Action: Recalculates re-identification risk on anonymized dataset

System: Displays updated risk assessment with comparison:

Risk Assessment Summary:

  • Original Risk: 0.21
  • Post-Anonymization Risk: 0.09
  • Risk Reduction: 57%
  • Status: WITHIN ACCEPTABLE THRESHOLD ✓

Options:

  • "View Risk Report" - Post-anonymization analysis
  • "View Sample Data" - Preview anonymized data
  • "View Effective Class Table" - Anonymization transformation summary

Step 13: Request Closure and Documentation

Actor: User

Action: Reviews final outputs and documentation

System: Generates complete audit trail including:

  • All user decisions and approvals
  • Anonymization rules applied
  • Risk calculations (before/after)
  • Data utility metrics

Actor: User

Action: Closes request

Button: "Close Request"

Final Status: "Completed Approved"

Agentic UX Design Patterns

Conversational Workflow Orchestration

The agent guides users through complex multi-step processes using natural language, autonomously determining the next appropriate step based on workflow state and user responses.

Key Features:

  • Proactive explanation of capabilities and process
  • Context-aware prompting at each decision point
  • Natural language interaction reduces learning curve

Progressive Disclosure

Information and options are revealed sequentially based on workflow progression, preventing cognitive overload.

Implementation:

  • Stage 1: Dataset identification
  • Stage 2: Data selection
  • Stage 3: Quasi-identifier detection
  • Stage 4: Context configuration
  • Stage 5: Risk assessment
  • Stage 6: Mitigation strategy (if needed)
  • Stage 7: Execution and verification

Transparent Execution Tracking

Left sidebar displays persistent workflow progress checklist:

  • ✓ Request Initiation
  • ✓ Risk Calculation
  • ✓ Risk Report Available
  • ✓ Anonymization Rule Generation
  • ✓ Re-Anonymization
  • ✓ Re-Anonymization Completed and Risk Report Available

Benefit: Users always know their position in the process and what remains.

Risk Visualization

Gauge-style visualization provides immediate visual feedback on data safety status:

  • Green zone: Risk below threshold (safe)
  • Yellow zone: Risk near threshold (caution)
  • Red zone: Risk exceeds threshold (action required)

Displays both calculated risk (0.09) and threshold (0.2) for clear comparison.

Glossary

  • Quasi-identifier: Data attribute that combined with others could identify individuals.
  • Re-identification risk: Probability that an individual can be uniquely identified in dataset.
  • Data utility loss: Measure of information reduction due to anonymization.
  • Risk threshold: Maximum acceptable re-identification risk for given sharing context.
  • Generalization: Replacing specific values with broader categories (e.g., age → age range).
  • Suppression: Removing or masking specific values while preserving structure.
  • K-anonymity: Privacy model ensuring each record is indistinguishable from k-1 others.
